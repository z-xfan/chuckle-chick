# 需求模块

本目录是 ChuckleChick 的需求讨论与决策记录中心。每项产品功能、行为变更或重要技术建设，都必须先在这里建立独立文档，再进入开发。

## 基本规则

- 一项需求对应一份文档，不把互不相关的需求混在同一文档中。
- 需求文档至少应明确背景、目标、范围、验收标准和状态。
- 状态为“草稿”或“讨论中”时，不进入主体代码开发。
- 只有状态为“已确认”的需求才可以开始开发。
- 实施中发生范围或验收标准变化时，先更新需求文档，再继续开发。
- 完成后补充实现位置和验证结果，保证需求与代码可以互相追溯。

## 状态流转

`草稿` → `讨论中` → `已确认` → `开发中` → `待验收` → `已完成`

如果需求不再实施，可将状态更新为 `已取消`，并记录原因。

## 文件命名

使用以下格式：

```text
REQ-<四位序号>-<简短英文名称>.md
```

例如：

```text
REQ-0002-desktop-app-foundation.md
```

序号在本目录内递增且不复用。新建需求时复制 [`TEMPLATE.md`](TEMPLATE.md)，不要直接修改模板文件。

## 开发准入检查

开始修改主体代码前，必须确认：

- 已存在对应需求文档；
- 文档状态为“已确认”或“开发中”；
- 范围和不做事项清晰；
- 验收标准可执行、可验证；
- 代码变更能够引用需求编号。
