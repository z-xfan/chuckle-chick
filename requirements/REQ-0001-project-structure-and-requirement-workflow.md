## 背景

项目将同时维护需求讨论资料、桌面应用主体代码和角色资源。为了避免需求只存在于口头讨论中，或在范围尚未明确时直接开发，需要建立统一的需求模块和开发准入规则。

## 目标

- 明确项目由需求文档与主体代码/资源两部分组成。
- 为每个需求保留可追溯的独立文档。
- 确保开发开始前已经明确需求范围和验收标准。

## 范围

### 包含

- 建立 `requirements/` 需求模块。
- 提供统一的需求文档模板、编号和状态规则。
- 在项目入口说明中明确需求先行的开发流程。

### 不包含

- 选择桌面应用技术栈。
- 创建桌面应用代码骨架。
- 实现任何产品功能。

## 需求说明

1. 每一项新需求必须在 `requirements/` 下创建独立文档。
2. 需求文档必须说明背景、目标、范围和可验证的验收标准。
3. 需求状态达到“已确认”后，才允许进入主体代码开发。
4. 需求变化必须先同步到文档，再调整实现。
5. 开发完成后，文档应记录实现位置与验证结果。

## 验收标准

- [x] 仓库存在独立的 `requirements/` 目录。
- [x] 需求模块包含使用说明和可复制的需求模板。
- [x] 根目录 README 明确项目的两部分内容与需求先行规则。
- [x] 模板包含状态、范围、验收标准、实施记录和验证记录。

## 实施记录

- `README.md`：增加项目组成与工作流程说明。
- `requirements/README.md`：定义需求管理规则、状态和开发准入检查。
- `requirements/TEMPLATE.md`：提供标准需求文档结构。
