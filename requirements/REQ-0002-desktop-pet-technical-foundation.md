# REQ-0002：Windows 桌面宠物技术方案

## 基本信息

- 状态：开发中
- 创建日期：2026-07-21
- 最后更新：2026-07-21
- 负责人：待定

## 背景

ChuckleChick 需要在 Windows 桌面显示一个透明背景的宠物。宠物能够播放不同动作和神态，并为后续与宠物相关的交互功能保留扩展空间。

仓库已经具备一套 v2 宠物资源：透明 PNG 动画图集、宠物元数据和图集描述。图集描述包含不同动作的帧位置、逐帧播放时长以及 16 个观察方向。

## 目标

- 选择适合长期维护的 Windows 桌面技术栈。
- 可靠支持透明、无边框、置顶的宠物窗口。
- 支持图集动画、宠物移动、鼠标交互和系统托盘。
- 将宠物能力与具体窗口框架解耦，便于后续增加互动、状态和插件式功能。
- 在视觉流畅度和常驻资源占用之间取得平衡。

## 范围

### 包含

- 桌面应用技术栈选型。
- 应用核心模块划分。
- 窗口、动画、行为、资源和配置的基础设计。
- 首个可运行版本的建议边界。

### 不包含

- 本需求阶段不编写应用代码。
- 不定义尚未提出的联网、AI、账号或云端能力。
- 不一次性设计完整插件系统。
- 不确定具体 UI 视觉稿。
- 不监听全局键盘输入。
- 不监听或支持游戏手柄输入。
- 不建设应用内自动更新能力。
- 不接入 Live2D运行时，不支持 Live2D模型格式。

以上四项是当前项目的明确技术边界，不只是首版延期项。未来确有需要时，必须创建独立需求文档，重新评估权限、安全、依赖、资源格式和发布成本后才能实施。

## 已确认基础范围

- 正式产品面向 Windows 10 22H2及 Windows 11，不要求发布 macOS或 Linux版本。
- 宠物窗口透明、无边框、不显示在任务栏，并默认置顶；用户可通过托盘关闭置顶。
- 用户可以拖动宠物；宠物能够播放待机和随机动作。
- 应用通过系统托盘提供显示/隐藏、设置和退出入口。
- 首版只加载本地宠物资源，不依赖网络服务。
- 暂不让宠物覆盖全屏游戏等独占场景。

## 技术路线对比

| 方案 | 优点 | 主要问题 | 结论 |
| --- | --- | --- | --- |
| C# + .NET 10 + WPF + 少量 Win32 | Windows 桌面能力成熟；透明异形窗口直接；C# 开发效率高；与 HWND、托盘、多屏和 DPI 能力结合方便；适合当前 PNG 图集 | WPF 只能在 Windows运行，Mac日常开发依赖虚拟机 | 未选择 |
| C# + .NET 10 + Avalonia + Windows 原生适配 | 可在 Mac直接运行大部分界面与动画；支持透明、无边框、置顶和托盘 | Windows特殊能力仍需平台适配；与参考项目技术路线不同 | 未选择 |
| C# + WinUI 3 | 微软当前推荐的新 Windows 原生 UI；现代控件与 Fluent 风格更好 | 桌面宠物依赖的逐像素透明异形窗口并不是它最顺手的场景；特殊窗口处理更复杂 | 可用于未来独立设置窗口，不作为宠物窗口首选 |
| Electron + TypeScript | Web UI 开发快，动画生态丰富 | 安装体积和常驻内存偏大；透明窗口存在平台限制；桌面底层行为仍需原生扩展 | 团队只有 Web 技术时才考虑 |
| Tauri 2 + Vue 3 + TypeScript + Rust + PixiJS | BongoCat已有成熟实践；Mac可直接开发调试；透明窗口、托盘和跨平台构建能力完整；相比 Electron更轻 | 同时维护前端、Rust和平台窗口适配，需要严格控制边界 | 已选择 |
| C++ + Win32/Direct2D | 控制力和性能上限最高 | 开发、调试和长期维护成本明显更高 | 性能实测证明托管方案不足时再评估 |
| Unity/Godot | 动画和行为系统能力强 | 对小型常驻桌面程序偏重；系统托盘、窗口穿透和桌面集成需要额外处理 | 暂不采用 |

## 参考项目：ayangweb/BongoCat

GitHub 上的 `ayangweb/BongoCat` 是一个成熟的跨平台互动桌宠，当前实现采用 Tauri 2、Vue 3、TypeScript、Rust、PixiJS 和 Live2D。它证明了“Web 前端负责渲染和设置界面，Rust 负责原生桌面能力”的方案可以用于 Windows、macOS 和 Linux 桌宠。

### 实现结构

- Tauri 创建两个窗口：透明无边框的宠物主窗口，以及普通的偏好设置窗口。
- 主窗口配置 `transparent`、`decorations: false`、`shadow: false`、`alwaysOnTop` 和 `skipTaskbar`。
- Vue 页面使用透明 Canvas；PixiJS 提供渲染循环，`easy-live2d` 加载 Live2D 模型、动作和表情。
- Rust 使用 `rdev` 全局监听键盘和鼠标，使用 `gilrs` 监听手柄，再通过 Tauri 事件发送给前端。
- 前端把按键、鼠标和手柄事件转换成 Live2D 参数、动作、表情和覆盖图片。
- 点击穿透使用 Tauri 的 `setIgnoreCursorEvents`，拖动使用 `startDragging`。
- 托盘、开机启动、全局快捷键、自动更新、文件访问和日志使用 Tauri 插件。
- Pinia 保存窗口尺寸、位置、缩放、透明度和模型设置；多显示器逻辑会把窗口限制回当前屏幕。

### 平台适配

- macOS 使用 `tauri-nspanel` 将主窗口转成浮动 `NSPanel`，隐藏 Dock 图标，并处理所有桌面空间和全屏辅助窗口。
- Windows 使用 `SetWindowPos` 和 `HWND_TOPMOST` 维护置顶状态，并单独控制任务栏可见性。
- Linux 主要使用 Tauri 通用窗口 API。

### 与 ChuckleChick 的差异

BongoCat 的核心资产是 Live2D 模型，需要 Cubism模型、纹理、动作、表情和参数驱动；ChuckleChick 当前是规则 PNG图集和逐帧时长数据，不需要引入 Live2D。若采用 Tauri，只需使用 Canvas或 PixiJS实现图集播放器和行为状态机。

BongoCat 的方案对 Mac日常开发最友好，也验证了透明窗口、穿透、托盘和跨平台分发。但是它同时引入 Node.js、前端构建链、Rust、WebView和多个 Tauri插件，工程栈比 Avalonia复杂。其 Windows置顶循环和宽泛文件权限也不应直接照搬，应按本项目最小权限和事件驱动原则重新实现。

### 对选型的影响

- 项目确认沿用 BongoCat的 Tauri、Vue、TypeScript、Rust和 Canvas分层路线。
- ChuckleChick使用 PixiJS播放现有 PNG图集，不引入 Live2D运行时和 Cubism资产格式。
- 无论采用哪种方案，BongoCat的“宠物窗口与设置窗口分离”“平台适配层”“模型资源包”和“事件驱动输入”值得复用。

## 推荐方案

技术路线已确认为 Tauri 2、Vue 3、TypeScript、Rust和 Canvas/PixiJS。项目借鉴 BongoCat的工程分层与平台处理方式，但根据 ChuckleChick的 PNG图集资产进行简化，不复制其 Live2D、全局键盘/手柄监听、更新服务或宽泛文件权限。

### 技术栈

- 桌面外壳：Tauri 2。
- 设置界面与应用状态：Vue 3、TypeScript、Pinia。
- 构建工具：Vite、pnpm。
- 宠物渲染：HTML Canvas与 PixiJS。
- 原生能力：Rust和必要的 Windows API适配。
- 配置与元数据：JSON，继续兼容 `pet.json` 与 `atlas.json`。
- 自动化测试：TypeScript核心逻辑单元测试、Rust单元测试、Windows窗口集成验收。
- 打包：Tauri Bundler；Windows首版优先生成 NSIS安装包，最终形式仍需确认。

### 实施原则

- 宠物主窗口和设置窗口分离，主窗口保持最小内容与最小权限。
- 动画状态机、资源解析和行为调度使用纯 TypeScript模块，不直接依赖 Vue组件。
- PixiJS只负责绘制与逐帧调度，不决定宠物行为。
- Rust只负责 Tauri生命周期、托盘、持久化入口和确实需要原生能力的窗口操作。
- Tauri命令与事件必须定义明确的数据结构，避免前端任意访问本地文件和系统能力。
- 不引入全局键盘监听、手柄监听、自动更新或 Live2D相关依赖、权限、代码和配置。
- 鼠标输入仅用于窗口拖动、点击和已经确认的宠物交互；若需要读取全局鼠标位置，必须在对应功能需求中单独说明权限与行为。

## 开发环境约束

当前主要开发设备是 Apple Silicon Mac，运行 macOS 15.6。Tauri、Vue、TypeScript、Rust和 PixiJS均可在 Mac直接开发，宠物动画、透明窗口、拖动和设置界面可以在 macOS运行调试。Windows仍负责平台特有能力和正式发布验收。

### Mac 可以完成

- 编辑和运行 Vue、TypeScript、Rust、Canvas/PixiJS、JSON及测试代码。
- 直接启动 Tauri macOS开发版，验证资源加载、动画、状态机、拖动和设置界面。
- 执行前端构建、TypeScript检查、Rust检查和跨平台核心测试。
- 处理角色资源、执行静态检查并维护项目文档。

### Mac 不能完成

- 最终确认 Windows WebView中的透明边缘、渲染性能、置顶、点击穿透和系统托盘行为。
- 验证 Windows 多显示器、DPI、任务栏、全屏应用和开机启动行为。
- 生成并签署最终 Windows安装包；正式发布构建应在 Windows x64环境或 Windows CI完成。

### 推荐开发组合

1. Mac 作为日常编辑和资源处理主机。
2. Mac安装 Node.js、pnpm、Rust、Tauri前置工具和 Xcode Command Line Tools，承担绝大部分日常开发。
3. Windows 11 ARM虚拟机只用于阶段性窗口功能验证，安装 Rust、Node.js和 Microsoft C++ Build Tools。
4. 使用 Windows x64 CI 或实体 Windows x64设备完成正式构建和最终兼容性验证。

前端核心逻辑不得依赖具体操作系统。Windows原生能力集中在 `src-tauri`的平台模块内，保证 Mac开发版可以使用等价或空实现运行。

### 当前 Mac 容量评估

当前设备为 Apple M4、24 GB 统一内存，检查时约有 220 GiB 可用磁盘空间。设备可以承载一套本地 Windows 开发环境，但需要控制同时运行的重量级应用。

#### 运行内存

| 场景 | Windows 虚拟机内存 | 整机内存建议 | 评价 |
| --- | ---: | ---: | --- |
| 日常 Tauri、Vue和 Rust开发 | 不需要虚拟机 | 16–24 GB | 当前设备可以流畅承担 |
| Windows 平台阶段性验证 | 8–10 GB | 24–32 GB | 当前设备适用；Mac端应关闭 Android Studio等非必要重量级应用 |
| Mac 与 Windows 两侧同时运行多个 IDE、浏览器和调试器 | 16 GB | 48 GB | 更舒适，但不适合当前 24 GB 设备长期这样使用 |

当前设备可以将 Windows 11 ARM虚拟机设为 8 GB；只有并行编译明显受限时再提高到 10 GB。macOS侧保留 14–16 GB，用于 Codex、编辑器、浏览器和系统缓存。

24 GB 统一内存不可后期升级。对于当前规模的桌面宠物项目不需要更换设备，但运行虚拟机时不建议同时开启 Android Studio、多个 Chromium 编辑器和大量浏览器标签页。

#### 阶段性零付费方案

在技术验证阶段，可以使用以下组合避免新增软件费用：

```text
VMware Fusion 或 UTM
+ Windows 11 Enterprise ARM64 90 天评估版
+ Rust stable与 Cargo
+ Node.js、pnpm和 Microsoft C++ Build Tools
```

微软当前提供 Windows 11 Enterprise ARM64 ISO 的完整 90 天评估版，安装不需要产品密钥。评估期结束后系统会显示非正版通知并每小时关机，因此只能作为原型与短期验证环境，不能视为长期免费的 Windows 许可证，也不应通过反复重装规避授权。

VMware Fusion 当前免费且更适合作为本项目的首选零付费虚拟机。UTM 也是完全免费开源的，但其 Windows 虚拟机目前缺少 GPU 加速，不适合用于透明窗口性能和渲染质量的最终判断。

GitHub Actions 的标准 Windows runner 可以免费承担编译和自动化测试：公开仓库免费，私有仓库使用账户每月包含的免费额度。它不能提供交互式桌面，不能替代拖动、透明、DPI、多屏和托盘的人工调试。

若 90 天后仍要保持零新增费用，长期合法方案是使用已经具备有效 Windows许可证的实体机、公司设备或远程设备。Tauri已经把大部分调试移到 Mac，但仍不能免除 Windows特有窗口行为的最终验证。

## 建议架构

前端核心与平台能力通过带类型的 Tauri命令和事件隔离：

```text
Vue 应用与设置界面
        |
        +-- pet-core       宠物状态、行为调度、动作切换
        +-- pet-assets     pet.json / atlas.json解析与校验
        +-- pet-renderer   PixiJS图集加载、裁剪与逐帧绘制
        +-- platform       Tauri命令和事件的 TypeScript封装
                            |
                            +-- Rust/Tauri生命周期与托盘
                            +-- 平台窗口适配
                            +-- 配置持久化与受控文件访问
```

目录名称可以在创建工程时按 Tauri惯例调整，但依赖方向必须保持：Vue组件调用核心模块，核心模块不能反向依赖 Vue、PixiJS或 Tauri；PixiJS不决定行为；Windows API不进入 TypeScript核心逻辑。

## 核心设计

### 宠物窗口

- 无边框、背景透明、尺寸跟随宠物显示尺寸。
- 默认置顶且不显示任务栏图标。
- 支持鼠标拖动，限制最终位置不完全移出可用屏幕。
- 正确处理 Windows 缩放比例和多显示器坐标。
- 首版不提供点击穿透；后续如需支持，必须建立独立需求并通过窗口适配层实现。
- 全屏应用检测、贴边和跨屏行为单独形成后续需求，不阻塞基础版本。

### 动画播放

- `PetAssetLoader` 读取并校验 `pet.json` 与 `atlas.json`。
- 加载图集后按网格切分并缓存帧，避免播放过程中重复解码图片。
- `AnimationPlayer` 按 `durationsMs` 推进帧，不假设所有帧时长相同。
- 动作播放与屏幕移动分离：动画负责“显示什么”，移动控制器负责“宠物在哪里”。
- 默认动作结束后回到 `idle`，是否循环由动作定义或运行规则决定。

### 行为系统

- 使用有限状态机管理 `Idle`、`Acting`、`Dragging`、`Moving` 等顶层状态。
- 使用行为调度器决定何时播放挥手、跳跃等动作。
- 用户输入事件优先于随机行为，避免拖动时被自动动作打断。
- 后续功能通过明确的命令或事件进入核心层，不让功能模块直接操纵窗口和图片控件。

### 配置与数据

- 用户设置保存在 Tauri解析的平台应用数据目录；Windows最终落在当前用户的本地应用数据目录。
- 首版配置包括宠物位置、显示比例、是否置顶、是否开机启动和当前宠物。
- 配置需要版本号和默认值，以便以后平滑迁移。
- 资源目录保持现有 `assets/pets/<pet-id>/` 约定。

## 首个版本建议边界

建议先交付一个技术验证版本：

1. 启动后正确显示透明宠物窗口。
2. 能读取现有黄鸡资源并播放 `idle` 及至少两个动作。
3. 支持拖动、置顶、托盘显示/隐藏和退出。
4. 记住宠物位置，并正确处理常见 DPI 与双屏场景。
5. 常驻 30 分钟无明显内存持续增长，动画无肉眼可见卡顿。

桌面行走、点击穿透、开机启动和完整设置界面拆成后续独立需求，避免首个版本同时验证过多问题。首版允许在待机动作之间随机选择已有动作，但不移动宠物窗口。看向鼠标功能后来通过 REQ-0003 单独确认并加入当前首版范围。

## 风险与验证重点

- 透明窗口在不同显卡、缩放比例和远程桌面环境中的渲染差异。
- 宠物图集原始尺寸较大，应确认切帧缓存后的内存占用。
- 高频移动整个窗口可能比只更新动画帧更昂贵，需要用实测确定刷新频率。
- 置顶窗口可能遮挡其他应用，需要提供快速隐藏和关闭置顶的入口。
- 点击穿透与宠物交互天然冲突，需要明确切换方式与默认行为。
- Apple Silicon 上的 Windows ARM 虚拟机适合功能开发，但不能替代 Windows x64 实机或 CI 的最终验证。
- 需要验证 Windows WebView中的透明 PNG边缘、PixiJS帧定时、内存占用和 Tauri点击穿透行为。
- 不直接复制 BongoCat每 16ms重设置顶状态或全文件系统权限；如通用 Tauri API不足，再引入最小范围的 Rust平台实现。
- 依赖和权限清单中不得出现全局键盘钩子、手柄库、Tauri updater插件或 Live2D/Cubism运行库。
